# Моделирование производственного цеха

Решение тестового задания: дискретно-событийное моделирование цеха с
последовательной обработкой изделий и динамическим распределением между
станками.

## Требования

- Компилятор C++ с поддержкой C++20 (gcc, clang, MinGW)
- CMake ≥ 3.14
## Сборка

Цель приложения — `workshop`; при включённой опции `WORKSHOP_BUILD_TESTS` (это
**значение по умолчанию**, `ON`) дополнительно собирается `workshop_tests`.

### С приложением и тестами (по умолчанию)

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build
```

### Только приложение (без тестов и без загрузки Google Test)

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DWORKSHOP_BUILD_TESTS=OFF
cmake --build build
```

Исполняемый файл будет создан как `build/workshop` (или `build/workshop.exe`
под Windows / MinGW).
### Под Windows c MinGW

```powershell
cmake -S . -B build -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Release
cmake --build build
```

Чтобы собрать без тестов, к первой команде добавьте
`-DWORKSHOP_BUILD_TESTS=OFF`.

## Запуск приложения

```bash
./build/workshop path/to/input.txt
```

Под Windows из PowerShell можно так: `./build/workshop.exe path/to/input.txt`.

Программа читает входной файл из аргумента командной строки и пишет журнал
событий в стандартный вывод.

## Запуск тестов

Нужна сборка **с** `WORKSHOP_BUILD_TESTS=ON` (см. выше). После `cmake --build`:

```bash
ctest --test-dir build
```

Либо запустить напрямую исполняемый файл набора тестов: `build/workshop_tests`
или `build\\workshop_tests.exe`.

## Формат входного файла

```
M N
T_{0,0}    T_{0,1}    ...  T_{0,N-1}
T_{1,0}    T_{1,1}    ...  T_{1,N-1}
...
T_{M-2,0}  T_{M-2,1}  ...  T_{M-2,N-1}
q_0  a_{0,0} a_{0,1} ... a_{0,q_0-1}
q_1  a_{1,0} a_{1,1} ... a_{1,q_1-1}
...
q_{N-1} a_{N-1,0} ... a_{N-1,q_{N-1}-1}
```

- `M`, `N` — число типов изделий и число станков (`1 ≤ M, N ≤ 100`)
- `T[i][j]` — время операции `i` на станке `j` (целое в `[0, 10000]`)
- `q_j` — длина начальной очереди станка `j`; затем `q_j` типов изделий
  от головы к хвосту, каждый в `[0, M-2]`

При нарушении формата программа печатает первую ошибочную строку (в
оригинальном виде) и завершается.

## Формат вывода

Сообщения выводятся в хронологическом порядке. Поддерживаются 5 типов:

| Сообщение         | Смысл                                                                 |
|-------------------|------------------------------------------------------------------------|
| `start t k i j`   | в момент `t` на станке `j` начата операция `i` для изделия `k`         |
| `finish t k i j`  | в момент `t` на станке `j` завершена операция `i` для изделия `k`      |
| `wait t k i j p`  | изделие `k` поставлено в очередь станка `j` (длина очереди до этого `p`) |
| `ready t k j`     | изделие `k` полностью готово (последний станок — `j`)                  |
| `stop t`          | все изделия обработаны                                                 |

В пределах одного момента времени порядок типов сообщений задаётся ТЗ:
`finish` → `start` → `wait` → `ready`.

## Особенности реализации

- **Порядок вывода при равном `t`.** Он сосредоточен в компараторе очереди событий (`Simulator.cpp`): сначала время, затем тип события (`EventKind`), затем монотонный `seq`. Новый вид сообщения можно легко встроить, выбрав для него место в шкале приоритетов относительно остальных.

- **Выбор станка за `O(N)` на шаг.** Для каждого станка поддерживается сумма прогнозируемых времён по текущему содержимому короба (`waitTimeSum`); она обновляется при `enqueue` / `dequeue`, без пересчёта длины очереди целиком при каждом завершении операции.

- **Парсинг.** Числа читаются через `std::from_chars` по `std::string_view`. Нигде нет лишних перекопирований строк и аллокаций на них.

- **Буферизованный вывод.** `LogWriter` накапливает текст в памяти и сбрасывает в `ostream` блоками, что снижает число низкоуровневых записей при длинной трассировке.

- **Тесты.** Присутствуют рукописные Google тесты, а так же реализован скрипт генерирующий стресс-тесты для бенчмарков.

## Алгоритм

Используется дискретно-событийное моделирование. Единственный тип события —
`FinishEvent` (момент завершения операции на конкретном станке). Все
остальные действия (`start`, `wait`, `ready`) — прямые следствия finish либо
начального заполнения цеха в `t = 0`.

Очередь событий — `std::priority_queue` с упорядочиванием по
`(время, тип события, seq)` для устойчивого порядка при совпадающем времени.

Для каждого станка хранится:

- `std::deque<int>` — FIFO-очередь с идентификаторами изделий;
- `waitTimeSum` — сумма `T[type(p)][j]` по всем `p` из очереди,
  обновляется инкрементально при добавлении/изъятии изделий, поэтому
  выбор станка для следующей операции выполняется за `O(N)`.

Сложность: `O(S · N)` по времени, где `S` — общее число изделий.
При `S ≤ 10^5` и `N ≤ 100` это ≤ `10^7` операций.

## Примеры

В каталоге `examples/` лежит набор тестов:

- `example1.txt` — пример из раздела 9 ТЗ (матрица `3 5 / 4 6`);
- `m3n1_pipeline.txt` — `M=3`, один станок, три изделия в очереди;
- `m2n2_two_items.txt` — `M=2`, два станка, по одному изделию;
- `m3n3_three_machines.txt` — три станка, по три изделия в начальной очереди (несколько `wait`);
- `example3_one_machine.txt` — единственный станок, последовательная обработка;
- `example4_m1.txt` — `M=1`: матрицы времени во входе нет (`M-1 = 0` строк), одна строка на каждый станок — только длина короба (здесь везде пусто, сразу `stop 0`);
- `bad_*.txt` — примеры некорректного формата для проверки обработки ошибок;
- `gen_stress.py` — генератор стресс-тестов произвольного размера
  (например `python examples/gen_stress.py --M 100 --N 100 --S 100000 --out examples/stress.txt`).

Запуск:

```bash
./build/workshop examples/example1.txt
```
